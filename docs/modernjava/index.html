Passing code with Behaviour Parameterization

Behavior parameterization is a software development pattern that lets you handle frequent requirement changes. In a nutshell, it means taking a block of code and making it available without executing it. This block of code can be called later by other parts of your programs, which means that you can defer the execution of that block of code. For instance, you could pass the block of code as an argument to another method that will execute it later. As a result, the method’s behavior is parameterized based on that block of code. For example, if you process a collection, you may want to write a method that

  • Can do “something” for every element of a list
  • Can do “something else” when you finish processing the list
  • Can do “yet something else” if you encounter an error

This is what Behaviour Parameterization refers to.


Coping with requirement changes

Imagine there is a HR application which wants to get a list of female employees from a list of all the employees.


First attempt

enum Gender {
	MALE,
	FEMALE
}

class Employee {
	int age;
	String name;
	String position;
	Gender gender;
	

	public Employee(int age, String name, String position, Gender gender) {
		super();
		this.age = age;
		this.name = name;
		this.position = position;
		this.gender = gender;
	}
	
	public int getAge() {
		return age;
	}
	public void setAge(int age) {
		this.age = age;
	}
	public String getName() {
		return name;
	}
	public void setName(String name) {
		this.name = name;
	}
	public String getPosition() {
		return position;
	}
	public void setPosition(String position) {
		this.position = position;
	}
	public Gender getGender() {
		return gender;
	}
	
	public void setGender(Gender gender) {
		this.gender = gender;
	}
}

Code for getting only female employees would be like

public static List<Employee> getOnlyFemaleEmployees(List<Employee> employees) {
	List<Employee> femaleEmployees = new ArrayList<>();
	for (Employee employee : employees) {
		if(Gender.FEMALE.equals(employee.getGender())) {
			femaleEmployees.add(employee);
		}
	}
	return femaleEmployees;
}


Second attempt: parameterizing the gender

Now HR changes her mind and asks for Male and Female employees separately. In the new code, gender is passed as parameter

public static List<Employee> filterEmployees(List<Employee> employees, Gender gender) {
	List<Employee> result = new ArrayList<>();
	for (Employee employee : employees) {
		if(gender.equals(employee.getGender())) {
			result.add(employee);
		}
	}
	return result;
}

Calling this method by

List<Employee> femaleEmployees = filterEmployees(employees,Gender.FEMALE);
List<Employee> maleEmployees = filterEmployees(employees,Gender.FEMALE);

Now HR wants all the employees who are aged above 40 years. In this is becoming irritating. Still the code is….

public static List<Employee> filterEmployees(List<Employee> employees, int age) {
	List<Employee> result = new ArrayList<>();
	for (Employee employee : employees) {
		if(age == employee.getAge()) {
			result.add(employee);
		}
	}
	return result;
}


Third attempt - filtering with every attribute

Now HR asks, all female employees or all the employees age greater than 40.. This makes code ugly.. We are doing flag programming here.

public static List<Employee> filterEmployees(List<Employee> employees, Gender gender, int age, boolean flag) {
	List<Employee> result = new ArrayList<>();
	for (Employee employee : employees) {
		if ((flag && employee.getGender().equals(gender)) || (!flag && age > employee.getAge())) {
			result.add(employee);
		}
	}
	return result;
}


Behaviour Parameterization

We need a better way to implement filtering of the employees. If we step back and see, all the filtering criteria is like a function which checks for some attribute of the Employee and employee is added to the result if the result is true. We can abstract it into an interface which has a method taking an Employee and returning a boolean. And concrete classes to implement this interface so that we can pass around that concrete class into filter method which uses that instance and checks for the filter criteria.

interface EmployeeChecker {
	boolean check(Employee employee);
}

class EmployeeFemaleChecker implements EmployeeChecker{

	@Override
	public boolean check(Employee employee) {
		return Gender.FEMALE.equals(employee);
	}
	
}

class EmployeeAgeChecker implements EmployeeChecker {

	@Override
	public boolean check(Employee employee) {
		return employee.getAge() > 40;
	}
	
}


Fourth attempt - filtering by abstract criteria

our modified filter with EmployeeChecker looks lik this

public static List<Employee> filterEmployees(List<Employee> employees, EmployeeChecker checker) {
	List<Employee> result = new ArrayList<>();
	for (Employee employee : employees) {
		if (checker.check(employee)) {
			result.add(employee);
		}
	}
	return result;
}

we can call this filter as below..

List<Employee> femaleEmployees = filterEmployees(employees,new EmployeeFemaleChecker());
List<Employee> agedEmployees = filterEmployees(employees,new EmployeeAgeChecker());

Now we have abstracted the behaviour in a way that you can write any filter condition just by implementing EmployeeChecker interface. For example you can write

  • EmployeeFemaleOrAgedAboveForty class
  • EmployeeMaleAndNameStartingWithLetterA class

There is a pattern for what we did just now. It’s called Strategy pattern. THIS WHOLE STUFF IS PASSING THE BEHAVIOUR TO A METHOD Now we have to tackle the problem of verbocity. Just to filter the collection we have to write so many classes. There should be an easier way to deal with it. Let’s see :).


Tackling Verbocity

The unnecessary overhead of creating too many classes somewhat is solved by Anonymous Classes.

Anonymous Classes

Anonymous classes are like the local classes (a class defined in a block) that you’re already familiar with in Java. But anonymous classes don’t have a name. They allow you to declare and instantiate a class at the same time. In short, they allow you to create ad hoc implementations.


Fifth attempt - using an anonymous class

Following code shows how to implement EmployeeChecker as anonymous class and use it for filtering

List<Employee> femaleEmployees = filterEmployees(employees, new EmployeeChecker() {
	@Override
	public boolean check(Employee employee) {
		return Gender.FEMALE.equals(employee.getGender());
	}
});

List<Employee> femaleAgedEmployees = filterEmployees(employees, new EmployeeChecker() {
	@Override
	public boolean check(Employee employee) {
		return employee.getGender().equals(Gender.FEMALE) && employee.getAge() > 40;
	}
});


Sixth attempt - using lambda expressions

Using lambda expressions

List<Employee> maleEmployees = filterEmployees(employees,
	(Employee employee) -> employee.getGender().equals(Gender.MALE));
List<Employee> femaleEmployeesBelow30 = filterEmployees(employees,
	(Employee employee) -> employee.getGender().equals(Gender.FEMALE) && employee.age < 30);


Seventh attempt - abstracting over List type

Now imagine you want to generalize this whole thing. You can use Generics along with the interface.

interface Predicate<T> {
	boolean test(T t);
}

public static <T> List<T> filter(List<T> list, Predicate<T> p) {
	List<T> result = new ArrayList<>();
	for(T e: list) {
		if(p.test(e)) {
			result.add(e);
		}
	}
	return result;
}

Now this can filter anything. like Integers, Apples, Employees etc,.

List<Employee> someEmployees = filter(employees, (Employee e) -> e.getAge() > 40);
List<Integer> someIntegers = filter(Arrays.asList(new Integer(1), new Integer(2), new Integer(3)),
	(Integer i) -> i.intValue() > 2);


Summary

Behavior parameterization is the ability for a method to take multiple different behaviors as parameters and use them internally to accomplish different behaviors.

Behavior parameterization lets you make your code more adaptive to changing requirements and saves on engineering efforts in the future.

Passing code is a way to give new behaviors as arguments to a method. But it’s verbose prior to Java 8. Anonymous classes helped a bit before Java 8 to get rid of the verbosity associated with declaring multiple concrete classes for an interface that are needed only once.

The Java API contains many methods that can be parameterized with different behaviors, which include sorting, threads, and GUI handling.



Lambda Expressions

After realizing that Anonymous Classes are too verbose for throwing around behavioural code. Because of the verbocity, developers do not like to use it. We know that Lamda Expressions are provided for this very reason, let’s learn them in step-by-step manner as usual.


Lambdas in a Nutshell

A Lambda Expression can be understood as a consise representation of Anonymous function that can be passed around. No name, has parameters, body and return type. Exceptions too can be thrown.

  • Anonymous - No name, less to code :).
  • Function - Because it’s not associated with any class and has parameters, body and return type and alos exceptions.
  • Passed Around - Can be passed around into a method or stored in a variable.
  • Consise - No need of too much boilerplate code like Anonymous classes.

Where and How to use Lambdas

You can use Lambdas in the place where a Functional Interface is expected.


Functional Interface

In a nutshell, a *functional interface* is an interface that specifies exactly one abstract method. You already know several other functional interfaces in the Java API such as Comparator and Runnable,.

Functional Interfaces already existed

public interface Comparator<T> {
	int compare(T o1, T o2);
}

public interface Runnable {
	void run();
}

public interface ActionListener extends EventListener {
	void actionPerformed(ActionEvent e);
}

public interface Callable<V> {
	V call() throws Exception;
}

public interface PrivilegedAction<T> {
	T run();
}

What can you do with functional interfaces? Lambda expressions let you provide the implementation of the abstract method of a functional interface directly inline and treat the whole expression as an instance of a functional interface (more technically speaking, an instance of a concrete implementation of the functional interface). You can achieve the same thing with an anonymous inner class, although it’s clumsier: you provide an implementation and instantiate it directly inline.


Function Descriptor

The signature of the abstract method of the functional interface describes the signature of the lambda expression. We call this abstract method a function descriptor. For example, the Runnable interface can be viewed as the signature of a function that accepts nothing and returns nothing (void) because it has only one abstract method called run, which accepts nothing and returns nothing (void).
